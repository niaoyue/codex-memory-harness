# Codex 外脑插件执行文档

> 历史设计说明：本文是项目早期从 0 到 1 的执行方案，用于保留架构来龙去脉。当前真实进度、下一步和 workspace routing 拆分，以 `.codex/harness/backlog/tasks.md` 与 `.codex/harness/backlog/workspace-routing-tasks.md` 为准。

## 1. 文档目的

本文档用于落地一套“以插件方式增强 Codex 四层能力”的执行方案，目标是为当前代理形态补齐以下能力：

1. 会话工作记忆
2. 外部混合检索
3. 提示层与上下文编排
4. 蒸馏到别的可训练目标

本文档只定义实施边界、阶段目标、架构拆分、验收标准与风险控制，不直接包含代码实现细节。

## 2. 背景与约束

### 2.1 背景

当前 Codex 具备较强的推理与工具调用能力，但长期稳定性仍主要依赖当次会话上下文。若要把其增强为“更稳定、可复用、可持续沉淀”的代理，需要在模型外部建设一层显式记忆系统。

### 2.2 当前约束

- 不修改底层模型权重。
- 优先采用插件化、本地可控、可逐步演进的方案。
- 面向自动调用，不依赖每次人工提醒。
- 先做最小可用版本，再逐步增强。
- 本仓库规则要求涉及任务编排时使用 `shrimp-task-manager`；早期执行时该工具不可用，因此采用本地 Markdown 文档作为保守降级方案。

## 3. 目标与非目标

### 3.1 目标

- 提供一个本地插件骨架，作为 Codex 外脑系统入口。
- 提供一个 MCP 记忆服务，统一暴露记忆资源与工具接口。
- 建立会话状态、项目记忆、证据包、蒸馏资产的最小存储结构。
- 实现任务开始前读记忆、任务进行中写状态、任务结束后沉淀资产的闭环。
- 支持后续按阶段逐步补齐检索、上下文编排、蒸馏管线。

### 3.2 非目标

- 不在第一阶段追求完整企业级知识平台。
- 不在第一阶段训练任何副模型。
- 不在第一阶段接入复杂远程服务。
- 不在第一阶段实现全自动高精度 rerank 或复杂向量检索。

## 4. 成功标准

达到以下条件时，可认为 MVP 成功：

1. 插件目录可被创建并保持结构清晰。
2. MCP 服务可读写当前任务状态与项目记忆。
3. Codex 可通过统一资源读取当前会话摘要、任务状态和项目决策。
4. 任务完成后，系统能自动生成一份结构化总结。
5. 总结可被回写为后续任务可复用的记忆资产。

## 5. 总体架构

目标架构如下：

```text
Codex
  -> Plugin Shell
    -> Hooks
    -> MCP Memory Server
    -> Context Builder
    -> Distillation Worker
    -> Local Storage
```

各组件职责如下：

### 5.1 Plugin Shell

- 作为插件入口与安装单元。
- 注册 hooks、skills、scripts、MCP 配置。
- 承载默认行为与本地配置。

### 5.2 Hooks

建议最小支持以下钩子：

- `on_session_start`
- `before_task`
- `after_tool`
- `before_response`
- `on_task_complete`

这些钩子负责自动触发记忆读取、状态更新、上下文压缩与任务沉淀。

### 5.3 MCP Memory Server

统一暴露显式记忆资源与工具接口。

最小资源建议：

- `memory://session/current`
- `memory://task/state`
- `memory://repo/profile`
- `memory://repo/decisions`
- `memory://distilled/patterns`

最小工具建议：

- `memory.get_context_pack`
- `memory.search`
- `memory.update_task_state`
- `memory.write_decision`
- `memory.write_summary`
- `memory.distill_task`

### 5.4 Context Builder

负责把上下文组织成稳定输入，而不是把所有历史原样塞给模型。

最小输入块：

1. 固定规则
2. 当前任务状态
3. 外部证据包
4. 最近工具发现
5. 当前用户请求

### 5.5 Distillation Worker

负责把高价值结果沉淀为可复用资产。

第一阶段只做：

- 任务总结
- 决策记录
- 高价值模式归档
- 技能草稿生成占位

## 6. 存储设计

第一阶段坚持本地、简单、可调试：

- `SQLite`：结构化状态、决策、索引元数据
- `JSONL`：事件日志
- `Markdown`：人工可读总结与蒸馏结果

建议目录：

```text
plugins/codex-memory/
  .codex-plugin/
  hooks/
  skills/
  scripts/
  mcp/
  storage/
    memory.db
    events.jsonl
    summaries/
    distilled/
```

## 7. 分阶段实施

### 阶段 0：脚手架与骨架

目标：

- 创建插件目录与基础清单文件。
- 创建 MCP 服务最小骨架。
- 明确存储目录与配置入口。

验收：

- 插件目录完整。
- 至少有一个可启动的 MCP 服务入口文件。
- 基础配置文件存在且结构可扩展。

### 阶段 1：会话工作记忆 MVP

目标：

- 定义任务状态结构。
- 实现读写 `task state`、`session summary`、`repo decisions`。
- 支持任务结束后的结构化总结回写。

验收：

- 可写入并读取当前任务状态。
- 可持久化决策记录。
- 可生成会话总结。

### 阶段 2：外部混合检索 MVP

目标：

- 实现代码优先的最小混合检索。
- 检索顺序先精确匹配、再全文检索、最后语义补充占位。

验收：

- 能对本地代码与文档生成证据包。
- 检索输出带来源、摘要、命中理由。

### 阶段 3：上下文编排 MVP

目标：

- 实现 `context pack` 构建逻辑。
- 控制注入预算，避免无差别拼接。

验收：

- 能生成稳定的上下文结构。
- 证据、状态、规则三类内容被显式分离。

### 阶段 4：蒸馏闭环 MVP

目标：

- 在任务完成后提取高价值资产。
- 回写到 `distilled/` 与结构化表中。

验收：

- 至少能产出任务总结、决策项、模式归档三类资产。

### 阶段 5：强化与自动化

目标：

- 完善 hooks 自动触发。
- 增强检索质量与冲突处理。
- 为后续接入副模型或更强检索系统预留接口。

验收：

- 系统可在日常任务中稳定复用。
- 失败时能优雅降级到无记忆模式。

## 8. 风险与控制

### 8.1 记忆污染

风险：

- 低质量总结和错误决策被长期保留。

控制：

- 区分“事实”“假设”“决策”“待确认项”。
- 默认只固化高置信结果。

### 8.2 上下文膨胀

风险：

- 注入内容过多，降低模型表现。

控制：

- 使用 `context pack`，而非原始全量历史。
- 对工具输出先摘要再注入。

### 8.3 自动化过重

风险：

- 每轮都做重型处理，导致响应慢。

控制：

- 在线只做轻量读写。
- 重型蒸馏与归档改为异步或任务结束后处理。

### 8.4 架构过度设计

风险：

- 在 MVP 阶段引入过多组件，拖慢落地。

控制：

- 先做本地单机方案。
- 先保证功能闭环，再考虑扩展。

## 9. 历史建议执行顺序

早期方案推荐按以下顺序推进：

1. 脚手架插件骨架
2. 定义状态模型与存储
3. 打通 MCP 记忆读写
4. 增加任务总结回写
5. 增加本地混合检索
6. 增加上下文编排
7. 增加蒸馏与模式归档

## 10. 历史下一步
