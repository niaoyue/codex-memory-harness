# Requirements Document

## Introduction

The repository has accumulated several Codex-generated planning and progress documents under `docs/`. Those files are useful for agent workflow, but they are not public user documentation. The formal home for durable Codex-generated requirements, design, task plans, PRDs, RFCs, change specs, and execution plans is `.codex/specs/<feature-slug>/`.

This spec migrates the project backlog and planning source of truth into `.codex/specs/backlog-governance/` while keeping short compatibility entries in `docs/` for existing links.

## Glossary

- **Backlog_Governance_Spec**: The canonical `.codex/specs/backlog-governance/` directory that owns this project's agent-facing backlog and planning records.
- **Canonical_Task_List**: `.codex/specs/backlog-governance/tasks.md`, migrated from `docs/codex-memory-plugin-task-list.md`.
- **Compatibility_Stub**: A short `docs/` file that points readers to the canonical spec location without preserving agent-owned planning state in `docs/`.
- **Planning_Source**: A durable requirements, design, task, execution plan, or progress document generated by Codex for implementation workflow.
- **Runtime_State**: `.codex/harness/tasks/`, `.codex/memories/`, databases, event logs, caches, and session artifacts that must not be treated as specs.

## Grill-Me Self-Check

The user corrected the target location from `.codex/harness/specs/` to `.codex/specs/`. That correction is blocking and must be reflected in implementation.

- **Resolved from repository rules:** `.codex/harness` is runtime harness state; durable generated specs belong under `.codex/specs`.
- **Resolved from existing spec:** `.codex/specs/codex-generated-documents/` already establishes the Kiro-like `requirements.md`, `design.md`, `tasks.md` convention.
- **Resolved from code search:** `unfinished_task_summary.py` currently defaults to `docs/codex-memory-plugin-task-list.md`, so migration must update the default reader or preserve a full legacy task list in `docs/`. The correct fix is to prefer the new canonical task list and fall back to the legacy path.
- **Resolved from HarnessTest dogfood and user clarification:** T59 uses the Codex SubAgent channel; it does not wait for an extra host SubAgent API. The repository must still avoid claiming plugin-owned independent SubAgent process management, and T59 remains visible until the Codex SubAgent dispatch/observation/checkpoint/receipt dogfood is documented as a repeatable workflow. T81, T83, and T87 still have host hook, branch integration, or upstream BMAD work that must not be claimed complete.

No user-facing open question blocks this migration.

## Requirements

### Requirement 1 - Canonical planning home

**User Story:** As a maintainer, I want generated planning and progress documents to live under `.codex/specs`, so that `docs/` stays focused on public and long-lived project documentation.

#### Acceptance Criteria

1. WHEN migrating Codex-generated planning documents, THE canonical output SHALL be `.codex/specs/backlog-governance/`.
2. WHEN a document is runtime state, memory, event log, cache, or harness task execution data, THE document SHALL NOT be migrated into `.codex/specs`.
3. WHEN existing public links point to the old `docs/` planning files, THE repository SHALL keep compatibility stubs that point to the canonical `.codex/specs` files.
4. IF a future agent is given `.codex/harness/specs` as the target for durable planning docs, THEN it SHALL correct the target to `.codex/specs` before editing.

### Requirement 2 - Backlog source of truth

**User Story:** As a maintainer, I want the unfinished task parser and human readers to use the same canonical task list, so that progress summaries remain accurate after migration.

#### Acceptance Criteria

1. WHEN no explicit task list path is supplied, THE unfinished task summary reader SHALL prefer `.codex/specs/backlog-governance/tasks.md`.
2. IF the canonical task list does not exist, THEN the reader SHALL fall back to `docs/codex-memory-plugin-task-list.md`.
3. WHEN `tasks.md` lists unfinished tasks, EACH unfinished task SHALL include status, recent progress or checkpoint, remaining acceptance, blocker, next step, and evidence source when available.
4. WHEN T59 is summarized, THE document SHALL distinguish the supported Codex SubAgent dispatch channel from unsupported plugin-owned independent agent process management, and SHALL keep T59 unfinished until repeatable dispatch/observation/checkpoint/receipt dogfood is documented.
5. WHEN T81, T83, or T87 are summarized, THE document SHALL distinguish locally verified adapter behavior from host-native hook, integration branch, or upstream BMAD work that remains blocked.

### Requirement 3 - Migrated document mapping

**User Story:** As a maintainer, I want the old planning files to have clear canonical successors, so that future agents do not create duplicate backlogs.

#### Acceptance Criteria

1. `docs/codex-memory-plugin-task-list.md` SHALL point to `.codex/specs/backlog-governance/tasks.md`.
2. `docs/codex-memory-plugin-execution-plan.md` SHALL point to `.codex/specs/backlog-governance/execution-plan.md`.
3. `docs/WORKSPACE_ROUTING_TASK_LIST.md` SHALL point to `.codex/specs/backlog-governance/workspace-routing-tasks.md`.
4. README or high-level docs that list the canonical planning source SHALL mention `.codex/specs/backlog-governance/`.

### Requirement 4 - Verification and review boundary

**User Story:** As a maintainer, I want this migration to be verified without committing runtime state, so that the repository remains clean and reviewable.

#### Acceptance Criteria

1. Verification SHALL include targeted tests for unfinished task summary path selection.
2. Verification SHALL include markdown diff checks and repository status review.
3. The intended commit boundary SHALL include only migrated specs, compatibility stubs, targeted code/tests, and directly related public references.
4. `.codex/shared/**`, `.codex/harness/tasks/**`, `.codex/memories/**`, caches, and generated runtime artifacts SHALL stay out of the commit boundary.
